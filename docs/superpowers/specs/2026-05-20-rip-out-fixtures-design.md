# Rip out fixtures

**Status:** approved
**Date:** 2026-05-20
**Author:** Dan Lourenco (with Claude)

## Motivation

Templates currently render against optional "fixture" data — a JSON object stored inline in the user-level registry and materialized to a temp file on every render. The same data can be overridden one-shot via `--data <file>` on `render` and `shot`. Three different missing-variable modes (`error`, `placeholder`, `empty`) decide what happens when a template references a variable not in the fixture.

In practice, the existing `previewMissingAs: 'placeholder'` behavior is sufficient on its own: undefined references render inline as `‹varName›` chevron spans, which is everything the user needs to visually inspect a template. The whole fixture system is dead weight — it adds:

- a temp-dir/materialize/cleanup dance in three places (server, VS Code panel, CLI)
- a per-render registry re-read in the dev server
- a Jackson-backed ISO-date coercion pass on the Java side
- a `--data` CLI flag and a `--missing` flag
- two fixture-specific error types
- documentation users have to read to understand a feature they don't need

This spec removes all of it. Templates render against `{}`; every variable becomes a `‹varName›` placeholder span; there is no knob.

## Non-goals

- Reworking the registry beyond removing one field
- Changing inline CSS, dev SSE refresh, VS Code preview shell, screenshot capture, or any rendering behavior other than the fixture path
- Backfilling pre-`v0.1.0` history into conventional-commit format

## §2 — Behavior changes

- **Templates always render against `{}`.** Every variable reference produces a `<span class="fmp-variable" style="...">‹varName›</span>` inline.
- **No mode toggle.** The `--missing` CLI flag and the `previewMissingAs` config key are gone. There is one behavior.
- **No data injection at any layer.** No `--data`, no registry `fixture` field, no temp-fixture-dir, no Java fixture-file argument.

- `.ftlh` templates used as test inputs (`hello.ftlh`, `numbers.ftlh`, `styled.ftlh`, `welcome.ftlh`, `errors/*.ftlh`, `include-test/templates/...`)
- `.json` fixture data co-located with them, plus the legacy `welcome.fixtures/` subdir

Action:
1. Rename `fixtures/` → `test-templates/`
2. Delete every `.json` file inside it
3. Delete `welcome.fixtures/`
4. Delete `dated.ftlh` (its only purpose was exercising ISO-date coercion, which no longer exists)

## §5 — Migration story for users

- Registry files with `fixture:` or `previewMissingAs:` keys → silently ignored on next load (loader already tolerates unknown fields).
- `--data <file>` / `--missing <mode>` invocations → fail with "unknown flag" from the CLI arg parser. Acceptable: it's a loud, immediate signal that the flag is gone.
- Single migration note in the `v0.1.0` `CHANGELOG.md` entry covering all three.
